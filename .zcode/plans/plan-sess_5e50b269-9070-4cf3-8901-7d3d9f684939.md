## Plan: 4-Way Provider Comparison (results-first, minimal code)

The user wants **comparison results**: does Grok hallucinate less than MiniMax/Claude/DeepSeek on the same screenshots? Not a provider framework. This plan gets to results with the smallest possible change.

### Step 1 ‚Äî Add Grok as a named provider (enables clean A/B, ~30 lines)

Without this, the harness has to juggle `process.env.OPENAI_BASE_URL_EXTRACTION` between runs to point Grok at x.ai vs real OpenAI ‚Äî fragile and confusing. Adding Grok as a named provider means `tagImage({extractionProvider: 'grok'})` just works, same as `minimax`/`claude`.

This is the MiniMax template (also OpenAI-compatible, also multimodal) applied to x.ai:
- `src/tagger.ts`: add `"grok"` to Provider union, `grokConfigForPass()` factory (reads `XAI_API_KEY`, `XAI_BASE_URL`, `XAI_AUTO_TAG_MODEL`), add to dispatch/resolve/hasVisionKey
- `src/env.ts` + `src/scripts/ui-server.ts`: add `grokKeyConfigured` to allowlists
- `.env`: move the existing `OPENAI_*_EXTRACTION=x.ai` block to `XAI_*` vars (the key is already there)

No new HTTP code ‚Äî `callOpenAICompatible` already handles x.ai's `/chat/completions` + `image_url` format.

### Step 2 ‚Äî Write the harness + run it (the actual deliverable)

**`scripts/grok-eval.mjs`** ‚Äî based on the existing `ab-eval-2.mjs` pattern, extended to 4 providers. Fixes the `providerOverride` ‚Üí `extractionProvider` bug from the old scripts.

Runs on ~12 screenshots (mix of known-hallucination cases + pattern diversity):
- Extraction A/B/C: Grok vs MiniMax vs Claude via `tagImage({extractionOnly:true, extractionProvider})`
- Critique A/B/C/D: fixed baseline extraction, critique through Grok vs MiniMax vs Claude vs DeepSeek via `generateCritique`

**Per-provider metrics:**
- Hallucination count: scan all prose for icon-only assertions, pixel measurements, self-referential evidence (using the sanitizer's own regexes ‚Äî shows what each provider *would* emit before the gates catch it)
- Pattern agreement: does the extraction patternType match the consensus?
- Critique specificity: word count, a11y risk count
- Latency

Outputs to `/tmp/grok-eval-results.json` + a console summary table.

### Step 3 ‚Äî Report findings

The output answers: which provider hallucinates least, which has the best extraction agreement, which writes the most specific critique, and at what latency/cost. That's the decision basis for the bulk retag provider choice.

### What gets committed
1. `feat(tagger): add Grok 4.5 as named provider` (Step 1 ‚Äî the enabler)
2. Run `scripts/grok-eval.mjs`, report the 4-way comparison table (Step 2 ‚Äî the deliverable)

No eval-set commit, no `generateCritique` fix, no UI changes ‚Äî those are follow-ups if the comparison justifies them. Get the data first.
